# soild 원칙

solid 원칙의 5가지 원칙

------------------------------


+ 1. SRP (Single Responsibility Principle) - 단일 책임 원칙
"어떤 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 한다."

개념: 하나의 클래스는 하나의 책임(기능)만 가져야 한다.

적용 예시: '유저 정보 관리' 클래스와 '데이터베이스 저장' 클래스를 분리한다. 이렇게 하면 DB 시스템이 바뀌어도 유저 비즈니스 로직은 수정할 필요가 없다.

+ 2. OCP (Open/Closed Principle) - 개방-폐쇄 원칙
"소프트웨어 엔티티는 확장에는 열려 있어야 하고, 변경에는 닫혀 있어야 한다."

개념: 기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있어야 한다.

적용 예시: 결제 시스템을 만들 때 인터페이스를 활용한다. 새로운 결제 수단(카카오페이, 네이버페이)이 추가되어도 기존의 핵심 결제 로직 코드는 수정하지 않고, 새로운 클래스를 추가(확장)하는 방식으로 구현할 수 있다.

+ 3. LSP (Liskov Substitution Principle) - 리스코프 치환 원칙
"프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 한다."

개념: 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.

적용 예시: 부모 클래스의 의도와 다르게 자식 클래스에서 메서드를 오버라이딩하여 예기치 않은 오류를 발생시키는 것을 방지해야 한다. 자식 클래스는 부모 클래스의 책임을 무시하거나 재정의해서는 안 된다.

+ 4. ISP (Interface Segregation Principle) - 인터페이스 분리 원칙
"클라이언트는 자신이 사용하지 않는 메서드에 의존 관계를 맺으면 안 된다."

개념: 하나의 방대한 범용 인터페이스보다는, 특정 클라이언트를 위한 여러 개의 구체적인 인터페이스가 낫다.

적용 예시: 복합기라는 거대한 인터페이스 하나를 만들기보다는, 프린터, 스캐너, 팩스 인터페이스로 잘게 쪼개어 불필요한 의존성을 줄이는 것이 좋다.

+ 5. DIP (Dependency Inversion Principle) - 의존성 역전 원칙
"고수준 모듈은 저수준 모듈의 구현에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다."

개념: 구체적인 클래스(구현체)보다 인터페이스나 추상 클래스에 의존해야 한다.

적용 예시: 운전자가 '아반떼'라는 구체적인 차종에 의존하는 것이 아니라, '자동차'라는 추상적인 개념에 의존하도록 설계한다. 그래야 나중에 '소나타'로 차를 바꿔도 운전(코드 실행)이 가능하다.
